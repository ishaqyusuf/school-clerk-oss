## Question

How should the setup and review footers compute and display counts, readiness, checked-row state, and disabled import reasons?

## Context

The user emphasized that status should update based on checked rows: checked rows are selected by default, the footer should show ready-for-import and needs-fix counts, and the start import button must not enable while any selected row needs fixing.

## Resolve

- Define setup footer counters before verification.
- Define review footer counters after verification:
  - total rows
  - checked rows
  - checked ready/executable rows
  - checked rows needing fixes
  - unchecked rows
  - skipped rows
- Decide whether unchecked attention rows block the Start Import button. User direction implies checked rows drive execution, so unchecked blockers should not disable batch import.
- Decide how imported rows and single-row imported rows affect counts.
- Define the primary disabled reasons:
  - no checked executable rows
  - checked row needs gender
  - checked row needs classroom
  - checked row needs action
  - checked row needs selected match candidate
  - active verification/execution/job polling
- Preserve current batch payload semantics: only checked executable rows are sent, skipped rows are omitted, async job is used for batch import.

## Expected Answer

A deterministic count and button-state specification that can be directly converted into selectors/memoized helpers in the implementation.

## Comments

> *This was generated by AI during Wayfinder issue commenting.*

Base review footer state only on checked rows. Show total rows, checked rows, checked executable rows, checked rows needing fixes, unchecked rows, and skipped rows. `Start import` should enable only when at least one checked row is executable and no checked row needs gender, classroom, action, or required match candidate selection. Unchecked attention rows should remain visible in their section but should not block import.
