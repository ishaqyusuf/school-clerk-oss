# Define Minimal Setup Screen Contract

## Question

What exactly belongs on the first student import screen, and what should be removed or compressed so it becomes the requested paste-first, low-noise setup experience?

## Context

The current setup screen has a header/workflow stepper, three boxed setup controls, a large textarea, warning alert details, sidebar summary cards, and a footer. The user wants this first screen to be very simple: text input, a compact flat horizontal defaults form before the input, a footer status area, and one proceed button.

## Resolve

- Decide whether the modal header/workflow labels stay, are visually minimized, or are removed from the import body.
- Define the flat horizontal defaults form:
  - import mode
  - classroom/fallback classroom
  - global gender
  - any currently required setup control that must remain available
- Decide exact footer counters:
  - raw non-empty lines
  - parsed student rows
  - parser warnings
  - rows/lines needing fixes before proceed
- Decide whether parser warnings show only as footer status on the first screen or can expand into a compact details popover/list.
- Define disabled/proceed states for empty input, missing classroom in single mode, ambiguous classroom headers, missing parser-required fields, and loading classroom/name-guide data.
- Preserve local draft persistence and existing parser behavior.

## Expected Answer

A concise setup-screen behavior spec with visible controls, removed controls, status copy, and proceed gating rules that can be implemented without another design conversation.

## Comments

> *This was generated by AI during Wayfinder issue commenting.*

Resolve this by making the setup screen intentionally sparse: a compact single-row defaults bar, the paste textarea as the dominant surface, and a sticky footer with parsed rows, lines needing fixes, and one `Proceed` action. Remove sidebar summary cards and long warning panels from the default view; warnings can be summarized in the footer with an optional compact details popover. Use parsed student rows as the primary count, but keep raw non-empty line count as secondary context when warnings exist.
